# Requirements Document — notifications-hook (S12)

## Introduction

S12 builds the **Notifications** screen and the mechanism behind it: Clavis installs (and cleanly removes) a desktop‑notification **hook** in `~/.claude/settings.json` for selected Claude Code lifecycle events, so you get a system notification when a task finishes, when Claude Code messages you, or when it runs a tool. Three toggles (Completion / General / Tool‑use) map to the Stop / Notification / PreToolUse events; each enabled toggle installs a Clavis‑marked hook entry that fires a desktop notification, and each row has a **Test** button that fires a sample notification immediately. Hook installation is surgical — it preserves all of the user's existing hooks and only adds/removes Clavis‑marked entries — and de‑fingerprinted (our own marker, no predecessor strings, no fixed port, no local server).

## Alignment with Product Vision

Realizes `product.md` Feature 10 (notifications wired through a hook the app installs and cleanly removes) and design checklist items **64–65**. It honors safe writes (preserve the user's hooks) and the de‑fingerprint rule (no `__ccmate__`, no port 59948, no PostHog).

## Requirements

### Requirement 1 — Notification hook install/remove (surgical, marked)

**User Story:** As a user, I want Clavis to add a notification hook without disturbing my existing hooks, so my setup stays intact.

#### Acceptance Criteria
1. The backend SHALL map the three notification kinds to events: **Completion → Stop**, **General → Notification**, **Tool‑use → PreToolUse**.
2. WHEN a kind is enabled THEN the backend SHALL add a Clavis‑marked hook entry to that event's array in `~/.claude/settings.json` `hooks` (a `command`‑type hook whose command fires a desktop notification and carries a recognizable Clavis marker, e.g. a `clavis-notify` token), preserving every other event and every existing entry; WHEN disabled THEN it SHALL remove **only** the Clavis‑marked entry for that event.
3. Writes SHALL be atomic and preserve all unknown keys; the user's own hooks (e.g. Stop/PreToolUse entries they already have) SHALL never be modified or removed.
4. The notification command SHALL be cross‑platform appropriate (Linux `notify-send`, macOS `osascript`, Windows PowerShell toast) — Linux is the primary verified target.
5. The current enabled state SHALL be derivable by scanning `settings.json` `hooks` for the Clavis‑marked entries (so the UI reflects reality even across restarts).

### Requirement 2 — Notifications screen

**User Story:** As a user, I want the Notifications screen from the design, so I can choose when Claude Code pings me.

#### Acceptance Criteria
1. The screen SHALL render a card with three rows, each = label + description + a **Test** button + a **toggle**:
   - **Completion notifications** — "Notify when Claude Code finishes a task" (Stop).
   - **General notifications** — "Notify when Claude Code sends a message" (Notification).
   - **Tool‑use notifications** — "Notify when Claude Code runs a tool" (PreToolUse).
2. WHEN a toggle changes THEN the corresponding hook SHALL be installed/removed (Req 1) and the toggle SHALL reflect success/failure (toast on error, no optimistic corruption).
3. WHEN **Test** is used THEN a sample desktop notification SHALL fire immediately (via the Tauri notification plugin since Clavis is running), requesting permission on first use.
4. The toggles SHALL initialize from the real installed state (Req 1.5); off‑Tauri (gallery) SHALL use demo state.

## Non-Functional Requirements

### Code Architecture and Modularity
- Backend: `core/notify_hook.rs` (read/derive state + install/remove the Clavis‑marked hook in `settings.json`, reusing `atomic_fs`/`settings`/`claude_json`) + `commands/notifications.rs`; the Tauri **notification plugin** for the Test action; DTOs in `model.rs`. Frontend: `src/screens/notifications/`, hooks in `queries.ts`. Add `tauri-plugin-notification` + its capability.
- The hook command is generated by one helper (per‑OS), and the marker is one constant.

### Performance
- Each toggle is a single parse/mutate/atomic‑write of `settings.json`.

### Security
- Only `settings.json` `hooks` is edited; no credential files, no `mcpOAuth`. The notification command contains no secrets. The marker/identity is `app.clavis` style — no predecessor strings, no fixed port, no background server.

### Reliability
- Surgical add/remove preserves the user's existing hooks; atomic writes; idempotent (enabling twice doesn't duplicate; disabling when absent is a no‑op). Malformed `hooks` degrades gracefully.

### Usability
- Plain labels matching the design; honest description that this writes a hook into `settings.json`; Test gives immediate feedback; per‑OS the command differs but the UX is identical.
